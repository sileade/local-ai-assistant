# Отчёт об аудите Scoliologic AI Console v4.6.0

**Дата аудита:** 13 апреля 2026 г.
**Аудитор:** Manus AI
**Версия компонента:** `sclg_ai_console.py` v4.6.0

## 1. Архитектурный обзор

`sclg_ai_console.py` представляет собой монолитный скрипт (3417 строк), реализующий автономного DevOps/SysAdmin агента. Архитектура построена вокруг концепции "Execute first, explain later" (сначала выполни, потом объясни), что отличает его от традиционных чат-ботов.

Система состоит из 21 класса, которые можно разделить на следующие логические блоки:

| Блок | Классы | Описание |
|------|--------|----------|
| **Оркестрация** | `SclgAI`, `SmartExecutor` | Главный цикл приложения и механизм предварительного выполнения команд на основе паттернов. |
| **Маршрутизация** | `ExpertRouter` | MoE (Mixture of Experts) классификатор, направляющий запросы к специализированным моделям (code, sysadmin, analysis и др.). |
| **Интеграция LLM** | `OllamaClient`, `ClaudeClient` | Клиенты для локального GPU-балансировщика и облачного API Anthropic с поддержкой квот и fallback-механизмов. |
| **Память и Контекст** | `DreamMemory`, `RequestCache` | Двухуровневая память (краткосрочная/долгосрочная) и кэширование запросов. |
| **Постобработка** | `ResponseCleaner`, `QualityChecker` | Очистка ответов от "AI-измов", удаление токсичных токенов, детекция отказов и зацикливаний. |
| **Интерфейс (UI)** | `OutputFormatter`, `Spinner`, `CommandRenderer`, `TypewriterEffect` | Визуальное оформление в стиле Claude Code: анимации загрузки, посимвольный вывод, форматирование таблиц и IP-адресов. |
| **Инфраструктура** | `NetworkScanner`, `SSHManager`, `InfraLearnerBridge` | Встроенные утилиты для сканирования сети, удалённого выполнения команд и интеграции с фоновым демоном мониторинга. |

## 2. Оценка возможностей (Capabilities Assessment)

### 2.1. Сильные стороны

Система демонстрирует высокий уровень адаптации к задачам системного администрирования:

**Превентивное выполнение (Smart Patterns)**
Агент способен распознавать 22 паттерна запросов (сеть, диски, процессы, Docker и др.) и автоматически выполнять соответствующие bash-команды *до* обращения к LLM. Это радикально снижает галлюцинации, так как модель работает с реальными данными системы.

**Многоуровневая отказоустойчивость**
Реализован сложный каскад fallback-механизмов:
1. Попытка использовать оптимальную локальную модель (например, `sclg-devops:27b`).
2. При неудаче — переключение на быструю резервную модель (`sclg-fast:7b`).
3. Если локальные модели недоступны или выдают отказ/мусор (детектируется через `QualityChecker`) — запрос уходит к Claude 3.5 Sonnet.
4. Если Claude недоступен или исчерпан лимит — пользователю выдаются сырые, но отформатированные результаты выполнения команд.

**Продвинутая очистка ответов**
`ResponseCleaner` эффективно удаляет типичные маркеры ИИ (вводные фразы, избыточные переходы, теги `<think>`), а также борется с зацикливаниями (repetition loops), характерными для локальных моделей при высоких значениях температуры.

**Интеграция с мониторингом**
`InfraLearnerBridge` позволяет агенту использовать знания, собранные фоновым демоном из Grafana и Prometheus, обогащая контекст системного промпта информацией об аномалиях и базовых метриках.

### 2.2. Ограничения и слабые места

Несмотря на мощную архитектуру, текущая реализация имеет ряд функциональных ограничений:

**Отсутствие потокового вывода (Streaming)**
Все вызовы API (как Ollama, так и Claude) выполняются в синхронном режиме без стриминга. Для сложных запросов пользователь видит только спиннер загрузки, пока не будет получен полный ответ, что ухудшает UX по сравнению с оригинальным Claude Code.

**Ограниченные возможности взаимодействия с ФС**
Агент не имеет встроенных инструментов для чтения, редактирования или создания файлов. Все операции с файлами должны выполняться через bash-команды внутри тегов `<cmd>`, что менее надёжно, чем использование специализированных функций (tool calling).

**Одношаговый цикл выполнения**
Агент обрабатывает теги `<cmd>` только один раз после генерации ответа. Если для решения проблемы требуется выполнить команду, проанализировать её вывод и затем выполнить следующую команду, текущая архитектура этого не позволяет (отсутствует полноценный Agent Loop).

**Отсутствие аутентификации и контроля доступа**
Любой пользователь, имеющий доступ к консоли, получает полные права на выполнение команд (включая `sudo`) и доступ ко всем хостам через `SSHManager`.

## 3. Выявленные уязвимости и баги

В ходе аудита исходного кода были обнаружены следующие проблемы, требующие исправления:

### Критические уязвимости

**Инъекция команд в SSHManager**
В методе `SSHManager.execute` (строка 2012) пароль передаётся в утилиту `sshpass` через форматирование строки:
```python
ssh_cmd = f"sshpass -p '{password}' ssh ..."
```
Если пароль содержит одинарную кавычку (`'`), это приведёт к выходу из контекста строки и возможности выполнения произвольного кода (Shell Injection). Рекомендуется использовать переменную окружения `SSHPASS` или передавать аргументы в виде списка через `subprocess.run(..., shell=False)`.

**Глобальный перехват исключений**
В коде присутствует более 30 блоков `except:`, перехватывающих абсолютно все исключения, включая `KeyboardInterrupt` и `SystemExit`. Это может привести к невозможности штатного завершения программы или скрытию критических ошибок памяти.

### Логические ошибки

**Неработающее регулярное выражение для детекции зацикливаний**
В `ResponseCleaner._cut_repetition_loops` (строка 532) используется регулярное выражение `(.{2,30?}){4,}`. Из-за отсутствия обратной ссылки (`\1`) оно совпадает с *любыми* 4+ группами символов, а не с повторяющимися подстроками. Тестирование показало, что данный метод (Method 1) является мёртвым кодом. Защита от зацикливаний работает только благодаря резервным методам (анализ триграмм и подсчёт строк).

**Двойная очистка ответов**
В методе `run()` (строка 3330) ответ очищается повторно:
```python
clean_response = self.cleaner.clean(response)
```
Хотя ответ уже был очищен ранее в методе `process_query` (строка 2531). Это создаёт избыточную нагрузку на процессор.

**Риск переполнения контекста**
В методе `_build_system_prompt` в системный промпт добавляются последние 4 сообщения из истории. Затем в `_get_ai_response` в массив сообщений добавляются последние 6 сообщений. Это приводит к дублированию контекста и риску превышения лимита токенов, особенно при наличии объёмного `data_context`.

## 4. Рекомендации по улучшению

Для дальнейшего развития Scoliologic AI Console рекомендуется:

1. **Рефакторинг безопасности:** Немедленно исправить уязвимость в `SSHManager`, переведя вызов `subprocess` на использование списка аргументов (`shell=False`) и передачу пароля через `env={"SSHPASS": password}`.
2. **Внедрение Tool Calling:** Заменить парсинг тегов `<cmd>` на использование нативного механизма вызова функций (Function Calling), поддерживаемого современными моделями (Claude 3.5, Qwen 2.5). Это позволит реализовать многошаговый агентский цикл.
3. **Поддержка стриминга:** Реализовать потоковое чтение ответов от Ollama и Claude для мгновенного отображения текста пользователю.
4. **Разделение монолита:** Разбить файл `sclg_ai_console.py` на модули (например, `core`, `ui`, `llm`, `tools`), что упростит поддержку и тестирование.
5. **Внедрение тестов:** Добавить unit-тесты для критических компонентов (`ExpertRouter`, `ResponseCleaner`, `SmartExecutor`), так как текущий CI/CD пайплайн проверяет только синтаксис.
